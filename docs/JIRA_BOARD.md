# Jira Board

Этот документ хранит правила работы с Jira-доской проекта. Jira остается
источником актуального состояния задач, а этот файл фиксирует договоренности
команды.

## Доска

| Поле | Значение |
|---|---|
| Настройки подключения | `.agents/skills/ledgerly-jira-board/config.local.yaml` |
| Последняя проверка документа | 14 июня 2026 |

## Где искать задачи

Актуальные задачи нужно искать только на Jira-доске, указанной в локальном
конфиге skill. Ветки Git, история коммитов и файлы репозитория не определяют
статус, приоритет или порядок задач.

### Порядок доступа через Atlassian MCP

Для автоматизированной работы использовать project skill
[`ledgerly-jira-board`](../.agents/skills/ledgerly-jira-board/SKILL.md).

1. Проверить доступ к проекту инструментом `getVisibleJiraProjects`:
   - значения подключения взять из `config.local.yaml`;
   - не выводить `cloud_id` без явного запроса.
2. Получить задачи инструментом `searchJiraIssuesUsingJql`.
3. Запрашивать как минимум поля `summary`, `status`, `priority`, `issuetype`,
   `parent`, `assignee` и `customfield_10019` (`Rank`).
4. Сортировать карточки по `Rank ASC`, если нужен порядок доски.
5. Не считать ошибку `403` от общего инструмента `search` отсутствием доступа
   к Jira. Rovo Search может быть не установлен, при этом прямые Jira API и
   JQL-запросы продолжают работать.

### Основные JQL-запросы

Текущая работа:

```jql
project = <PROJECT_KEY> AND statusCategory = "In Progress" ORDER BY Rank ASC
```

Все незавершенные задачи проекта:

```jql
project = <PROJECT_KEY> AND statusCategory != Done ORDER BY Rank ASC
```

Следующие задачи конкретного эпика:

```jql
project = <PROJECT_KEY> AND parent = <EPIC_KEY> AND statusCategory != Done ORDER BY Rank ASC
```

Чтобы определить следующую задачу, сначала нужно найти активный эпик, затем
выполнить запрос по его `parent` и взять первую незавершенную карточку по
`Rank ASC`. Эпик не считается следующей реализационной задачей; выбирать нужно
его дочернюю задачу.

## Назначение

Доска используется для планирования и отслеживания разработки Ledgerly:

- пользовательских возможностей;
- исправлений ошибок;
- технических задач и рефакторинга;
- изменений архитектуры и инфраструктуры;
- исследовательских задач.

## Актуальное состояние

Список колонок, активный спринт и задачи должны браться непосредственно из
Jira. Они не продублированы здесь, чтобы документ не становился устаревшей
копией доски.

На 14 июня 2026 чтение проекта и задач доступно через Atlassian MCP с помощью
прямых Jira API и JQL-запросов.

## Workflow

После подключения доступа зафиксировать реальные колонки доски:

| Колонка | Статусы Jira | Критерий перехода |
|---|---|---|
| Уточнить по доске | Уточнить по доске | Уточнить по доске |

Для каждой колонки важно описать не только название, но и условие, при котором
задача может перейти дальше. Это снижает число задач с неоднозначным статусом.

## Требования к задачам

Каждая задача должна содержать:

- понятный результат в заголовке;
- описание контекста или проблемы;
- критерии приемки;
- тип задачи;
- приоритет;
- исполнителя, когда задача взята в работу;
- связи с зависимыми и блокирующими задачами;
- ссылку на техническое решение или документацию, если они существуют.

### Рекомендуемый шаблон описания

```markdown
## Контекст
Почему нужна эта задача.

## Результат
Что должно измениться для пользователя или системы.

## Критерии приемки
- [ ] Проверяемое условие 1
- [ ] Проверяемое условие 2

## Технические заметки
Ограничения, зависимости и важные решения.
```

## Типы задач

| Тип | Когда использовать |
|---|---|
| Epic | Крупная цель, объединяющая несколько самостоятельных задач |
| Story | Пользовательская возможность с измеримым результатом |
| Task | Техническая или продуктовая работа без отдельного дефекта |
| Bug | Исправление поведения, которое не соответствует требованиям |
| Sub-task | Часть работы, не имеющая самостоятельной ценности вне родительской задачи |

Названия и доступность типов необходимо сверить с фактической конфигурацией
проекта.

## Приоритет

| Уровень | Значение |
|---|---|
| Highest | Блокирует выпуск, работу системы или критический пользовательский сценарий |
| High | Существенно влияет на ближайшую цель или устраняет серьезный риск |
| Medium | Обычная плановая задача |
| Low | Улучшение без срочной продуктовой или технической необходимости |
| Lowest | Идея или необязательная оптимизация |

## Правила ведения

1. Jira является источником истины по статусу, исполнителю и приоритету задач.
2. Задача переводится в работу только после уточнения результата и критериев
   приемки.
3. Блокировки отмечаются связями между задачами и поясняются комментарием.
4. Существенные решения переносятся из комментариев в документацию проекта.
5. Завершенная задача должна соответствовать критериям приемки и содержать
   ссылки на связанный pull request или изменения.
6. Устаревшие задачи закрываются или получают пояснение, почему остаются в
   бэклоге.

## Обновление документа

Документ нужно пересматривать при изменении:

- колонок или статусов workflow;
- типов задач и обязательных полей;
- правил приоритизации;
- процесса планирования и завершения задач;
- адреса, ключа проекта или ID доски.

При обновлении следует менять дату последней проверки в разделе «Доска».
